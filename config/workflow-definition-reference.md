# Alert-Triggered Workflow — Reference (build manually in Kibana)

This is a reference description only — **not** a deployable Workflow definition file. The Workflow itself is built by hand in Kibana's **Workflows** UI during the webinar, per spec (detection rule authoring and Workflow authoring are both intentionally manual — see `config/README.md` for why).

## Trigger

On new alert generated by the manually-authored AI/ES\|QL password-spraying detection rule (see `config/ai-detection-rule-prompt.md`).

## Steps (in order)

1. **Create case** — new case for the triggering alert. Use a descriptive title/template referencing the password-spraying scenario.
2. **Attach the triggering alert to the case** — link the rule that fired to the case created in step 1.
3. **Add an AI-generated analysis comment to the case** — use the case's AI assistant/analysis action to summarize the alert as a case comment, including the targeted-username list (`targeted_users`) and attempt count so the analyst sees the spray's blast radius without opening the alert.
4. **Run the response action** — trigger `runscript --script="block-spray-source.ps1"` (parameterised with the alert's `source.ip`, and optionally the `targeted_users` list) to block the offending IP at the Windows Firewall and disable the targeted accounts.
5. **Add a final summary comment to the case** — a closing comment noting the actions taken (case created, alert attached, source IP blocked, accounts disabled) so the full response is visible in the case timeline.

## Explicit exclusions

- **No external notification integration.** A Slack notification step was considered during design and deliberately dropped so the Workflow stays self-contained (no external connector/credentials required for the demo). Do not add a Slack (or any other outbound notification) step — the workflow ends at step 5, the final case comment.

## Acceptance criterion satisfied

This Workflow, executed end-to-end, satisfies **spec Acceptance Criteria #8**: the alert-triggered Workflow creates a case, attaches the triggering alert, adds an AI analysis comment, blocks the source IP via the runscript response action, and adds a final summary comment — all visible in the Cases UI.
