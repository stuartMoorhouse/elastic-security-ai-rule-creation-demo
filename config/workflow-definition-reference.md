# Alert-Triggered Workflow — Reference (build manually in Kibana)

This is a reference description only — **not** a deployable Workflow definition file. The Workflow itself is built by hand in Kibana's **Workflows** UI during the webinar, per spec (detection rule authoring and Workflow authoring are both intentionally manual — see `config/README.md` for why).

## Trigger

On new alert generated by **either**:

- The manually-authored AI/ES\|QL detection rule (see `config/ai-detection-rule-prompt.md`), or
- The prebuilt **"Suspicious Microsoft HTML Application Child Process"** rule

Both rules fire against the same simulated process tree (`forfiles → mshta → powershell → curl.exe → wscript/reg.exe`), so the trigger condition should match on either rule name/ID, not just one.

## Steps (in order)

1. **Create case** — new case for the triggering alert(s). Use a descriptive title/template referencing the ClickFix/LOLBin scenario.
2. **Attach the triggering alert(s) to the case** — link whichever rule(s) fired (AI rule and/or the prebuilt Mshta rule) to the case created in step 1.
3. **Add an AI-generated analysis comment to the case** — use the case's AI assistant/analysis action to summarize the alert(s) and process tree as a case comment.
4. **Isolate the host** — trigger an Elastic Defend response action to isolate the Windows VM that raised the alert.
5. **Add a final summary comment to the case** — a closing comment noting the actions taken (case created, alert(s) attached, host isolated) so the full response is visible in the case timeline.

## Explicit exclusions

- **No external notification integration.** A Slack notification step was considered during design and deliberately dropped so the Workflow stays self-contained (no external connector/credentials required for the demo). Do not add a Slack (or any other outbound notification) step — the workflow ends at step 5, the final case comment.

## Acceptance criterion satisfied

This Workflow, executed end-to-end, satisfies **spec Acceptance Criteria #9**: the alert-triggered Workflow creates a case, attaches the triggering alert(s), adds an AI analysis comment, isolates the host, and adds a final summary comment — all visible in the Cases UI.
